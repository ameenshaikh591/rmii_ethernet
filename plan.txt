# FPGA IPv4/UDP Offload Extension

## Hardware and Software Integration Plan for the Existing RMII NIC

## 1. Project objective

Extend the existing Basys 3 RMII Ethernet NIC into a small hardware IPv4/UDP endpoint supporting:

* Static local IPv4 configuration
* ARP request generation and reply handling
* ARP replies to requests for the FPGA
* A small hardware ARP cache
* Unfragmented IPv4 packets with no IP options
* IPv4 header checksum generation and verification
* UDP transmit with checksum set to zero
* UDP receive without checksum verification
* Two hardware-bound UDP receive ports
* Three RX entries per UDP socket
* One shared four-entry TX queue
* Descriptor-based payload movement through the existing AXI DMA infrastructure
* Polling-based MicroBlaze software APIs:

  * `udp_init`
  * `udp_socket_open`
  * `udp_socket_close`
  * `udp_send`
  * `udp_recv`

ARP resolves IPv4 next-hop addresses to Ethernet addresses. IPv4 provides the header checksum and fragmentation fields, while UDP provides source/destination ports, length, and an optional checksum; a transmitted UDP checksum of zero indicates that no checksum was generated for IPv4.

The implementation is deliberately limited to:

* IPv4 only
* Ethernet II only
* No VLAN tags
* No IPv4 options
* No IPv4 fragmentation or reassembly
* No ICMP
* No DHCP
* No multicast
* No TCP
* No UDP checksum generation or verification
* One outstanding ARP resolution at a time
* In-order TX descriptor processing

These restrictions are appropriate for the intended FPGA endpoint and keep the design bounded.

# 2. Four important architectural corrections

## 2.1 The status manager must not orchestrate packet transmission

The AXI-Lite status manager should remain a register bank. It should not:

* Launch descriptor DMA reads
* Wait for ARP
* Construct headers
* Stream payloads
* Control TX frame generation

Those responsibilities belong to a dedicated `udp_tx_engine`.

The persistent condition:

```text
TX_HEAD != TX_TAIL
```

is the work request. No AXI-Lite write pulse needs to be preserved or retried.

---

## 2.2 The Ethernet MAC should continue owning the Ethernet header

Do not make the ARP or UDP engines generate complete Ethernet frames if the existing MAC TX already generates:

* Preamble
* SFD
* Destination MAC
* Source MAC
* EtherType
* Ethernet padding
* FCS
* Interpacket gap

Instead, submit this metadata to the existing FIFO/MAC infrastructure:

```text
destination MAC
EtherType
Layer-2 payload length
Layer-2 payload stream
```

The FIFO contents should be:

* For ARP: the 28-byte ARP packet
* For UDP: the 20-byte IPv4 header, 8-byte UDP header, and application payload

This avoids duplicating Ethernet-header and FCS logic.

---

## 2.3 Use wrap-aware queue pointers

A plain modulo index cannot distinguish empty from full when all queue entries are usable.

Keep the existing 32-bit HEAD/TAIL registers, but define only bits `2:0`:

```text
bit 2       wrap/generation bit
bits 1:0    entry index
bits 31:3   zero
```

For an RX queue of depth three, valid pointer values progress as:

```text
0b000  index 0, wrap 0
0b001  index 1, wrap 0
0b010  index 2, wrap 0
0b100  index 0, wrap 1
0b101  index 1, wrap 1
0b110  index 2, wrap 1
```

For the four-entry TX queue:

```text
000 → 001 → 010 → 011 → 100 → 101 → 110 → 111 → 000
```

Definitions:

```text
empty =
    head == tail

full =
    head.index == tail.index
    and head.wrap != tail.wrap
```

This allows all three RX slots and all four TX slots to be used without a hidden “completed but unpublished” packet state.

The VHDL and C software must use the same pointer-advance helper.

---

## 2.4 Cache maintenance is separate from memory barriers

If the MicroBlaze data cache is enabled and the DMA packet region is cacheable:

* `udp_send()` must flush the completed TX entry before advancing `TX_TAIL`.
* `udp_recv()` must invalidate the selected RX entry after observing a new tail and before reading metadata or payload.

A memory barrier controls ordering but does not by itself write dirty cache lines back or invalidate stale lines. AMD provides MicroBlaze cache APIs for flushing and invalidating address ranges.

The simplest implementation choices are:

1. Place the DMA packet region in uncached memory; or
2. Add cache flush/invalidate calls inside the UDP software library.

The Codex agent must inspect the MicroBlaze cache configuration before deciding.

# 3. Top-level architecture

```text
                         AXI-Lite
MicroBlaze ──────────────────────────────────────────────┐
                                                       │
                                              +--------v---------+
                                              | UDP Status       |
                                              | Manager          |
                                              | registers only   |
                                              +--+----+----+-----+
                                                 |    |    |
                       configuration/pointers    |    |    |
                                                 |    |    |
                      +--------------------------+    |    +------------------+
                      |                               |                       |
              +-------v--------+              +-------v--------+      +-------v------+
              | UDP RX Engine  |              | UDP TX Engine  |      | ARP Manager  |
              | parser + queue |              | descriptor     |<---->| cache + ARP  |
              | controller     |              | scheduler      |      | resolution   |
              +-------+--------+              +-------+--------+      +-------+------+
                      |                               |                       |
                 AXI writer                      AXI reader              ARP producer
                      |                               |                       |
                      +-------------------------------+-----------------------+
                                                      |
                                               +------v-------+
                                               | TX Frame     |
                                               | Arbiter      |
                                               +------+-------+
                                                      |
                                               +------v-------+
                                               | TX FIFO      |
                                               | Manager      |
                                               +------+-------+
                                                      |
                                                  MAC TX
                                                      |
                                                    RMII

RMII → MAC RX → RX asynchronous FIFO → UDP RX Engine
```

Clock domains:

* RMII MAC RX/TX: PHY reference clock domain
* Protocol engines, AXI reader/writer, status manager, ARP manager: AXI clock domain
* CDC occurs only through the existing asynchronous FIFOs and existing synchronized control handshakes

Avoid placing ARP, IPv4, or UDP parsing in the RMII clock domain.

# 4. Register map

The previously agreed queue map remains:

```text
0x00  DMA_BASE_ADDR       R/W

0x04  RX_QUEUE_0_CONFIG   R/W
0x08  RX_QUEUE_0_HEAD     R/W
0x0C  RX_QUEUE_0_TAIL     R

0x10  RX_QUEUE_1_CONFIG   R/W
0x14  RX_QUEUE_1_HEAD     R/W
0x18  RX_QUEUE_1_TAIL     R

0x1C  TX_QUEUE_HEAD       R
0x20  TX_QUEUE_TAIL       R/W
```

## 4.1 RX queue configuration

```text
RX_QUEUE_n_CONFIG

bit 31       ENABLE
bits 30:16   reserved
bits 15:0    bound local UDP port
```

The name `CONFIG` is preferable to `SOCKET`, because the register stores the bound UDP port. The software socket handle is simply queue index `0` or `1`.

## 4.2 Required network configuration

The queue map alone is insufficient unless IPv4 configuration is compiled into VHDL generics.

The design must know:

* Local IPv4 address
* Subnet mask
* Default gateway IPv4 address
* Local MAC address

For a practical MicroBlaze-controlled endpoint, append:

```text
0x24  LOCAL_IPV4_ADDR     R/W
0x28  SUBNET_MASK         R/W
0x2C  DEFAULT_GATEWAY     R/W
```

The local MAC can remain an existing VHDL generic or constant if it is already fixed in the NIC. Otherwise add two MAC-address registers later.

If the register map must remain exactly as originally listed, the three IPv4 values must instead be top-level VHDL generics. Runtime configuration is strongly preferable because the FPGA may be attached to different subnets.

A write to any network-configuration register should generate an internal pulse that invalidates the ARP cache.

# 5. DMA memory layout

Use exactly 1500 bytes per entry.

```text
DMA_BASE +     0   RX socket 0 entry 0
DMA_BASE +  1500   RX socket 0 entry 1
DMA_BASE +  3000   RX socket 0 entry 2

DMA_BASE +  4500   RX socket 1 entry 0
DMA_BASE +  6000   RX socket 1 entry 1
DMA_BASE +  7500   RX socket 1 entry 2

DMA_BASE +  9000   TX entry 0
DMA_BASE + 10500   TX entry 1
DMA_BASE + 12000   TX entry 2
DMA_BASE + 13500   TX entry 3

DMA_BASE + 15000   end of reserved region
```

Because 1500 is divisible by four, every entry remains 32-bit aligned.

Do not infer a general-purpose multiplier for address generation. Queue depths are fixed and tiny. Use a VHDL function or case statement:

```text
entry index 0 → offset 0
entry index 1 → offset 1500
entry index 2 → offset 3000
entry index 3 → offset 4500
```

The socket-1 and TX base offsets are constants. This should synthesize as small add/mux logic, not a DSP multiplier.

## 5.1 Common entry organization

```text
offset 0x000–0x00F   metadata, 16 bytes
offset 0x010–0x5CF   payload, up to 1472 bytes
offset 0x5D0–0x5DB   reserved, 12 bytes
```

A 1500-byte IPv4 packet with a 20-byte IPv4 header and 8-byte UDP header carries at most 1472 bytes of UDP application data.

## 5.2 TX metadata

```text
word 0:
    destination IPv4 address

word 1:
    bits 15:0   destination UDP port
    bits 31:16  source UDP port

word 2:
    bits 15:0   payload length
    bits 31:16  reserved/flags

word 3:
    reserved
```

The source port is copied into the descriptor by `udp_send()`. It should not be recovered from a live socket configuration later, because the socket could be closed or rebound after the descriptor was queued.

## 5.3 RX metadata

```text
word 0:
    source IPv4 address

word 1:
    bits 15:0   source UDP port
    bits 31:16  destination UDP port

word 2:
    bits 15:0   payload length
    bits 31:16  flags/reserved

word 3:
    reserved
```

The wire protocol is always big-endian. Shared-memory descriptor words should use the MicroBlaze system’s native 32-bit convention. Centralize all packing and unpacking in VHDL and C helper functions; do not cast arbitrary C structures directly onto network headers.

# 6. Status manager entity

Suggested entity:

```text
ws_udp_status_manager
```

Responsibilities:

* AXI-Lite reads and writes
* Hold software-owned registers:

  * DMA base
  * network configuration
  * RX queue configuration
  * RX heads
  * TX tail
* Hold hardware-owned registers:

  * RX tails
  * TX head
* Advance hardware-owned pointers on explicit commit pulses
* Generate ARP-cache flush pulse on network configuration changes

It must not start DMA transactions.

## 6.1 Status manager hardware interfaces

Outputs:

```text
dma_base_addr
local_ipv4_addr
subnet_mask
default_gateway
socket_enable[1:0]
socket_port[1:0]
rx_head_ptr[0:1]
rx_tail_ptr[0:1]
tx_head_ptr
tx_tail_ptr
network_config_changed
```

Inputs:

```text
rx_commit_valid
rx_commit_socket

tx_release_valid
```

On `rx_commit_valid`, advance the selected hardware RX tail using the depth-three pointer helper.

On `tx_release_valid`, advance TX head using the depth-four pointer helper.

The RX and TX engines should latch their current pointer before beginning a transaction. They must not repeatedly use a live pointer throughout a multi-cycle operation.

# 7. TX frame submission interface

The TX FIFO manager should accept a generic Layer-2 frame submission:

```text
frame_req_valid
frame_req_ready

frame_dest_mac
frame_ethertype
frame_l2_payload_length

frame_data_valid
frame_data_ready
frame_data[31:0]
frame_valid_bytes
frame_last

frame_abort
```

`frame_l2_payload_length` excludes:

* Destination/source MAC fields
* EtherType
* FCS

It includes:

* 28 bytes for ARP
* `20 + 8 + UDP payload length` for IPv4/UDP

The FIFO manager owns two buffer states:

```text
FREE
FILLING
READY
SENDING
```

A producer first reserves a free buffer through `frame_req_valid/ready`. Once granted, the producer owns that fill channel until:

* The final stream word is accepted; or
* `frame_abort` is asserted

The MAC TX remains responsible for Ethernet minimum-length padding. An ARP payload is only 28 bytes, so the MAC adds the required Ethernet padding after the ARP payload. Padding is not included in the ARP, IPv4, or UDP lengths.

---

# 8. TX frame arbiter

Suggested entity:

```text
ws_tx_frame_arbiter
```

Producers:

1. ARP manager
2. UDP TX engine

Priority:

```text
ARP > UDP
```

This avoids delaying an ARP request needed by a UDP descriptor waiting for resolution.

The arbiter is frame-granular and non-preemptive.

States:

```text
A_IDLE
A_GRANT_ARP
A_GRANT_UDP
```

Rules:

* Choose a producer only when the FIFO manager can reserve a buffer.
* Hold the grant until accepted `frame_last` or `frame_abort`.
* Never interleave words from two frames.
* ARP has priority only when selecting a new frame; it cannot interrupt a UDP frame already being filled.

# 9. ARP manager

Suggested entity:

```text
ws_arp_manager
```

The ARP manager should contain three logical pieces:

1. ARP cache
2. Address-resolution FSM
3. ARP transmit FSM

These may reside in one VHDL entity but should be separate processes or clearly separated logic.

## 9.1 ARP cache

Recommended first version:

```text
4 fully associative entries
```

Each entry contains:

```text
valid
IPv4 address, 32 bits
MAC address, 48 bits
```

Replacement:

```text
round-robin replacement pointer
```

No expiration timer is required initially. Invalidate the entire cache on:

* Reset
* Local IP change
* Subnet-mask change
* Gateway change

Cache updates should occur from validated ARP packets when:

* An ARP request targets the local FPGA IP
* An ARP reply targets the local FPGA
* The sender IP matches the currently pending resolution

A valid ARP request contains useful sender-IP/sender-MAC information, so it may satisfy a pending resolution even though its opcode is “request.”

## 9.2 Resolution interface

Input from the UDP TX engine:

```text
resolve_req_valid
resolve_req_ready
resolve_ip
```

Output:

```text
resolve_rsp_valid
resolve_rsp_mac
resolve_rsp_failed
```

Support one outstanding resolution request.

## 9.3 Address-resolution FSM

```text
R_IDLE
R_LOOKUP
R_REQUEST_PENDING
R_WAIT_REQUEST_SENT
R_WAIT_REPLY
R_RESOLVED
R_FAILED
```

### `R_IDLE`

Accept a resolution request and latch the requested next-hop IP.

### `R_LOOKUP`

Search all ARP cache entries.

* Hit: return MAC and enter `R_RESOLVED`
* Miss: create an ARP-request transmit request

### `R_REQUEST_PENDING`

Wait until the ARP transmit FSM accepts the request.

### `R_WAIT_REQUEST_SENT`

Wait until the ARP request has been completely buffered into a TX FIFO. Start the retry timer only after this point.

### `R_WAIT_REPLY`

Wait for a validated ARP event whose sender IP equals the pending IP.

* Match: update cache and return resolved MAC
* Timeout: retry
* Retry limit reached: return failure

Recommended generics:

```text
ARP_RETRY_INTERVAL_CYCLES
ARP_MAX_RETRIES
```

Use short values in simulation and realistic values in hardware.

An ARP timeout must not block the TX queue forever. On failure, the UDP TX engine drops that TX descriptor and advances TX head.

## 9.4 ARP RX event interface

Input from the RX parser:

```text
arp_event_valid
arp_event_opcode
arp_sender_mac
arp_sender_ip
arp_target_mac
arp_target_ip
```

The event is emitted only after the complete Ethernet frame has passed MAC/FCS/error validation.

## 9.5 ARP request/reply behavior

For a valid ARP request targeting `LOCAL_IPV4_ADDR`:

* Learn the sender mapping
* Queue an ARP reply
* Reply destination MAC = requester MAC
* Reply target MAC = requester MAC
* Reply target IP = requester IP

For a valid ARP reply targeting the local FPGA:

* Learn the sender mapping
* Complete a matching outstanding resolution

Ignore malformed ARP packets and ARP packets for unrelated protocols or target addresses.

## 9.6 ARP transmit FSM

```text
ATX_IDLE
ATX_SELECT
ATX_FRAME_REQUEST
ATX_STREAM
ATX_DONE
```

Maintain:

```text
one pending ARP reply slot
one pending resolver-generated ARP request flag
```

Prioritize replying to an incoming ARP request over sending a new resolution request.

If another incoming ARP request arrives while the one-entry pending reply slot is occupied, dropping the newer request is acceptable for version one; the requester will normally retry. A two-entry pending-reply FIFO can be added later if desired.

The ARP payload generator emits exactly 28 bytes:

```text
hardware type       Ethernet
protocol type       IPv4
hardware length     6
protocol length     4
opcode              request or reply
sender MAC
sender IP
target MAC
target IP
```

For an ARP request:

```text
Ethernet destination MAC = FF:FF:FF:FF:FF:FF
ARP target MAC            = 00:00:00:00:00:00
ARP target IP             = next-hop IP
```

For an ARP reply:

```text
Ethernet destination MAC = requester MAC
ARP target MAC            = requester MAC
ARP target IP             = requester IP
```

# 10. UDP TX engine

Suggested entity:

```text
ws_udp_tx_engine
```

Responsibilities:

* Detect pending TX descriptors
* Read TX metadata
* Validate metadata
* Compute the next-hop IP
* Request ARP resolution
* Build the IPv4 and UDP headers
* Request a TX FIFO through the frame arbiter
* Stream header words
* Request and stream the application payload
* Release the TX descriptor after the full Layer-2 payload has been buffered

## 10.1 TX FSM

```text
T_IDLE
T_META_REQ
T_META_RECV
T_VALIDATE
T_NEXT_HOP
T_RESOLVE_REQ
T_RESOLVE_WAIT
T_IP_CHECKSUM
T_FRAME_REQ
T_HEADER_STREAM
T_PAYLOAD_REQ
T_PAYLOAD_STREAM
T_FRAME_FINISH
T_RELEASE
T_DROP
```

## 10.2 `T_IDLE`

Check:

```text
TX_HEAD != TX_TAIL
```

If unequal:

* Latch the current TX head pointer
* Calculate the descriptor address
* Begin metadata read

## 10.3 `T_META_REQ`

Issue a 16-byte AXI read at:

```text
DMA_BASE
+ 9000
+ tx_entry_offset(head.index)
```

## 10.4 `T_META_RECV`

Capture four metadata words.

The AXI reader should support a mode where its returned data is consumed by the TX engine instead of the FIFO manager.

## 10.5 `T_VALIDATE`

Check:

* Payload length is `0–1472`
* Destination port is nonzero
* Source port is nonzero
* Destination IP is supported unicast IPv4
* Local IP configuration is valid
* Entry address does not overflow the 32-bit address space

For version one, reject:

* `0.0.0.0`
* `255.255.255.255`
* Multicast destinations
* Local-IP loopback to the FPGA itself

Invalid descriptors enter `T_DROP`. Since there is no completion-status register in version one, the descriptor is silently discarded and TX head advances.

## 10.6 `T_NEXT_HOP`

Compute:

```text
if (destination_ip AND subnet_mask)
   = (local_ip AND subnet_mask)
then
    next_hop_ip = destination_ip
else
    next_hop_ip = default_gateway
```

The final IPv4 destination always remains the original destination IP.

Only the Ethernet destination MAC corresponds to the next hop.

If the destination is off-subnet and the gateway address is zero or invalid, drop the descriptor.

## 10.7 `T_RESOLVE_REQ`

Request `next_hop_ip` from the ARP manager.

Do not reserve a TX FIFO while waiting for ARP.

## 10.8 `T_RESOLVE_WAIT`

Wait for:

* Resolved destination MAC; or
* ARP resolution failure

On failure, go to `T_DROP`.

## 10.9 `T_IP_CHECKSUM`

Construct the fixed 20-byte IPv4 header.

Recommended fields:

```text
Version/IHL          0x45
DSCP/ECN             0
Total length         20 + 8 + UDP payload length
Identification       incrementing 16-bit counter
Flags/offset         DF=1, MF=0, fragment offset=0
TTL                  64
Protocol             17
Header checksum      calculated
Source IP            local FPGA IP
Destination IP       descriptor destination IP
```

Use a small sequential checksum accumulator over ten 16-bit words instead of a large combinational adder tree. The IPv4 header checksum covers the IPv4 header only.

The UDP header is:

```text
Source port
Destination port
UDP length = 8 + payload length
UDP checksum = 0
```

A zero UDP checksum indicates that the transmitter did not generate one for IPv4.

## 10.10 `T_FRAME_REQ`

Request a TX FIFO frame with:

```text
destination MAC    resolved next-hop MAC
EtherType          0x0800
L2 payload length  20 + 8 + payload length
```

Wait until the frame arbiter and FIFO manager accept the request.

## 10.11 `T_HEADER_STREAM`

Emit the 28-byte IPv4/UDP header in network byte order.

Seven 32-bit words can hold the complete header.

Do not assert `frame_last` on the final header word unless UDP payload length is zero.

## 10.12 `T_PAYLOAD_REQ`

For a nonzero payload, issue an AXI read from:

```text
descriptor address + 16
```

with the exact application payload length.

## 10.13 `T_PAYLOAD_STREAM`

Forward AXI read data into the already-reserved TX FIFO.

The existing final-word valid-byte mechanism must preserve payload lengths not divisible by four.

On AXI read error:

* Assert `frame_abort`
* Discard the partially filled FIFO buffer
* Enter `T_DROP`

## 10.14 `T_FRAME_FINISH`

Wait until the FIFO manager confirms the complete frame payload has been committed and the buffer is `READY`.

## 10.15 `T_RELEASE`

Pulse `tx_release_valid`.

The status manager advances TX head.

This is the correct point to release the software TX entry because all metadata and payload bytes are now buffered internally. The MAC may still be transmitting the frame.

## 10.16 `T_DROP`

For:

* Invalid metadata
* Missing gateway
* ARP timeout
* AXI read error
* Internal frame-buffer error

Abort any reserved TX buffer, then advance TX head so one bad descriptor cannot permanently block the ring.

# 11. RX MAC and asynchronous FIFO

The MAC RX remains responsible for:

* Preamble/SFD handling
* Frame boundaries
* FCS validation
* Producing a byte stream
* Reporting frame errors

Recommended FIFO entry:

```text
data[7:0]
last
error
```

The parser must receive:

* Destination MAC
* Source MAC
* EtherType
* Layer-2 payload

If the current MAC strips the Ethernet header, either:

1. Stop stripping destination/source/EtherType; or
2. Carry those fields as sideband metadata associated with the frame

The Codex agent must inspect the existing MAC behavior before modifying it.

## 11.1 RX FIFO overflow

Ethernet cannot be backpressured. FIFO overflow must terminate the logical frame cleanly.

Recommended behavior:

1. Reserve one FIFO location for an abort marker by treating `almost_full` as unavailable for normal data.
2. If the FIFO reaches `almost_full` during a frame:

   * Enqueue `{last=1, error=1}`
   * Stop writing the current frame
   * Continue consuming RMII until physical end-of-frame
3. Do not begin writing the next frame until the abort marker has been committed.

Fallback if an abort marker cannot immediately be written:

* Set `abort_pending`
* Drop all incoming data
* Insert the abort marker as soon as space becomes available
* Do not admit a new frame before the marker

This prevents a truncated frame prefix from being concatenated with the next frame.

# 12. UDP RX engine

Suggested entity:

```text
ws_udp_rx_engine
```

For resource efficiency, combine these functions in one entity:

* Ethernet parser
* ARP parser
* IPv4 parser
* UDP parser
* Socket demultiplexer
* RX queue reservation
* AXI writer orchestration
* RX queue commit decision

The AXI writer remains a generic data mover and should not understand ARP, IPv4, UDP, or sockets.

## 12.1 RX FSM

```text
RX_IDLE
RX_ETH_HEADER
RX_DISPATCH

RX_ARP_BODY
RX_ARP_DRAIN
RX_ARP_COMMIT

RX_IPV4_HEADER
RX_UDP_HEADER
RX_SOCKET_SELECT
RX_WRITE_REQ
RX_UDP_PAYLOAD
RX_IP_DRAIN
RX_WAIT_WRITE
RX_COMMIT

RX_DROP
```

Maintain a sticky per-frame flag:

```text
frame_bad
```

Any input entry with `error=1` sets `frame_bad`.

## 12.2 `RX_IDLE`

Wait for the first byte of a new frame.

Initialize:

* Byte counters
* Header checksum accumulator
* Captured metadata
* `frame_bad = 0`

## 12.3 `RX_ETH_HEADER`

Read 14 bytes:

```text
destination MAC, 6
source MAC, 6
EtherType, 2
```

Accept frames whose destination MAC is:

* The local FPGA MAC
* Broadcast

Other frames enter `RX_DROP`.

## 12.4 `RX_DISPATCH`

```text
EtherType 0x0806 → RX_ARP_BODY
EtherType 0x0800 → RX_IPV4_HEADER
anything else    → RX_DROP
```

VLAN-tagged frames are unsupported and dropped.

# 13. ARP receive states

## 13.1 `RX_ARP_BODY`

Read the fixed 28-byte ARP payload and capture:

```text
hardware type
protocol type
hardware length
protocol length
opcode
sender MAC
sender IP
target MAC
target IP
```

Validate:

```text
hardware type   = Ethernet
protocol type   = IPv4
hardware length = 6
protocol length = 4
opcode          = request or reply
```

Do not notify the ARP manager yet.

## 13.2 `RX_ARP_DRAIN`

ARP Ethernet frames contain padding after the 28-byte ARP payload. Consume entries until `last`.

An ARP event may be committed only if:

* The complete ARP body was received
* The frame ended normally
* No FIFO/MAC/FCS error occurred
* The packet passed ARP field validation

## 13.3 `RX_ARP_COMMIT`

Send one structured event to the ARP manager.

For an ARP request, the ARP manager decides whether the target IP is local and whether to reply.

For an ARP reply, the ARP manager validates the target and resolves any pending request.

Return to `RX_IDLE`.

# 14. IPv4 receive states

## 14.1 `RX_IPV4_HEADER`

Parse exactly 20 bytes.

Capture:

```text
Version
IHL
Total length
Identification
Flags and fragment offset
TTL
Protocol
Header checksum
Source IP
Destination IP
```

Calculate the one’s-complement checksum incrementally over the full header, including the received checksum. A correctly formed header folds to all ones.

Require:

```text
Version              = 4
IHL                  = 5
Total length         >= 28
Total length         <= 1500
Protocol             = 17
Destination IP       = local FPGA IP
More Fragments       = 0
Fragment Offset      = 0
IPv4 checksum valid
```

The fragment offset and MF flag belong to IPv4. Packets failing the unfragmented requirement are dropped.

The DF flag may be either zero or one on receive.

If any validation fails, enter `RX_DROP`.

## 14.2 `RX_UDP_HEADER`

Read eight bytes:

```text
source port
destination port
UDP length
UDP checksum
```

Validate:

```text
UDP length >= 8
UDP length = IPv4 total length - 20
UDP payload length <= 1472
destination port != 0
```

The source port may be zero; UDP defines the source port as optional. The software should still record the received value. RFC 768’s user interface model returns data along with the source address and source port.

Ignore the UDP checksum field regardless of whether it is zero or nonzero. Document this clearly as a limitation.

## 14.3 `RX_SOCKET_SELECT`

Compare destination port against:

```text
socket 0 enabled and port match
socket 1 enabled and port match
```

Accept only an exact one-hot match.

* No match: drop
* Both match because of erroneous duplicate software configuration: drop
* Exactly one match: latch socket index

Then evaluate queue full using the wrap-aware pointers.

If full:

* Consume and drop the frame
* Do not start the AXI writer

If space exists:

* Latch the current tail pointer
* Calculate the entry address
* Continue to `RX_WRITE_REQ`

## 14.4 `RX_WRITE_REQ`

Request the AXI writer with:

```text
entry address
16-byte RX metadata
expected maximum payload length
```

The metadata contains the parsed:

* Source IP
* Source port
* Destination port
* Payload length

The parser may stall its FIFO reads while waiting for the writer to become ready.

## 14.5 `RX_UDP_PAYLOAD`

Forward exactly `UDP length - 8` bytes to the AXI writer.

The AXI writer should expose ready/valid backpressure.

If the Ethernet frame ends before the expected payload:

* Assert writer abort
* Mark frame bad
* Enter the appropriate drain/finalization state

If an RX FIFO error marker is encountered:

* Assert writer abort
* Mark frame bad

For a zero-length UDP application payload, skip directly to `RX_IP_DRAIN` after the metadata write request.

## 14.6 `RX_IP_DRAIN`

After consuming the exact IPv4 total length, continue consuming any Ethernet padding until `last`.

Do not write Ethernet padding into the UDP payload buffer.

At `last`, latch whether the complete frame was valid:

```text
frame_complete_good =
    not frame_bad
    and exact IPv4/UDP byte counts observed
```

## 14.7 `RX_WAIT_WRITE`

Wait for the AXI writer to report:

* Successful completion
* Aborted transaction completion
* AXI write error

The queue tail must not be advanced merely because the payload bytes were accepted from the parser. All AXI writes must complete successfully.

## 14.8 `RX_COMMIT`

Before committing, recheck:

* Selected socket remains enabled
* Selected socket remains bound to the same destination port

This prevents a packet from becoming visible after `udp_socket_close()` or a rapid close/rebind operation.

Commit only if:

```text
frame_complete_good
and AXI writer succeeded
and socket configuration still matches
```

Then pulse:

```text
rx_commit_valid
rx_commit_socket
```

The status manager advances that queue’s tail.

If any condition fails, do not advance the tail. Bytes may have been written into the unused slot, but the slot remains software-invisible and may be overwritten by the next accepted packet.

## 14.9 `RX_DROP`

Consume entries until `last`.

The parser must never return to `RX_IDLE` before the rejected frame terminates, or the next frame may be misinterpreted.

# 15. Generic AXI writer requirements

The existing writer should be adapted, not replaced unnecessarily.

Suggested interface:

```text
write_req_valid
write_req_ready
write_base_addr
write_metadata
write_payload_length

payload_valid
payload_ready
payload_data
payload_valid_bytes
payload_last

write_abort

write_done
write_error
write_aborted
```

Suggested logical states:

```text
W_IDLE
W_METADATA
W_PAYLOAD
W_FLUSH_PARTIAL
W_WAIT_RESPONSES
W_DONE
W_ERROR
W_ABORTED
```

On abort:

* Stop accepting new payload bytes
* Complete or drain already-issued AXI transactions
* Do not hang waiting for the originally expected byte count
* Report `write_aborted`

The RX engine controls queue publication, so an aborted or failed writer transaction never advances the RX tail.

# 16. Generic AXI reader requirements

The TX engine needs two read modes:

```text
metadata read
payload stream read
```

Suggested interface:

```text
read_req_valid
read_req_ready
read_addr
read_length
read_mode

read_data_valid
read_data_ready
read_data
read_valid_bytes
read_last

read_done
read_error
```

For metadata mode, the UDP TX engine captures the returned words.

For payload mode, data is passed through the TX frame arbiter into the selected FIFO.

Only one TX descriptor is processed at a time, so no tag or transaction ID is required initially.

# 17. MAC TX behavior

The existing MAC TX should consume per-buffer metadata:

```text
destination MAC
EtherType
Layer-2 payload length
```

It should generate:

```text
preamble
SFD
destination MAC
source MAC
EtherType
FIFO payload
zero padding if Layer-2 payload < 46 bytes
FCS
interpacket gap
```

For UDP:

```text
FIFO payload =
    IPv4 header
    UDP header
    application payload
```

For ARP:

```text
FIFO payload =
    ARP packet
```

The FIFO’s `last` marks the end of the real Layer-2 payload, before Ethernet padding.

# 18. Software API mapping

Recommended API:

```c
typedef int udp_socket_t;

typedef struct {
    uint32_t source_ip;
    uint16_t source_port;
    uint16_t destination_port;
    uint16_t payload_length;
    uint16_t flags;
} udp_info_t;

typedef struct {
    uintptr_t dma_base;
    uint32_t local_ip;
    uint32_t subnet_mask;
    uint32_t default_gateway;
} udp_config_t;

int udp_init(const udp_config_t *config);

udp_socket_t udp_socket_open(uint16_t local_port);

int udp_socket_close(udp_socket_t socket);

int udp_send(
    udp_socket_t socket,
    const void *payload,
    uint16_t payload_length,
    uint32_t destination_ip,
    uint16_t destination_port
);

int udp_recv(
    udp_socket_t socket,
    void *payload,
    uint16_t payload_capacity,
    udp_info_t *info
);
```

## 18.1 `udp_init`

Sequence:

1. Validate that DMA base is 32-bit aligned.
2. Disable both socket queues.
3. Program DMA base.
4. Program local IP, subnet mask, and gateway.
5. Set each RX head equal to its current tail.
6. Set TX tail equal to TX head.
7. Flush or invalidate the full packet region if required by the cache configuration.

## 18.2 `udp_socket_open`

Sequence:

1. Reject local port zero.
2. Reject duplicate binding.
3. Find a disabled queue.
4. Ensure queue config is disabled.
5. Read that queue’s tail.
6. Write head equal to tail.
7. Write port and set enable bit last.
8. Return socket ID `0` or `1`.

## 18.3 `udp_socket_close`

Sequence:

1. Validate socket ID.
2. Clear the socket enable bit.
3. Read current tail.
4. Write head equal to tail.
5. Clear the software socket state.

The RX engine’s commit-time configuration recheck prevents an in-flight packet from being published after close.

## 18.4 `udp_send`

Sequence:

1. Validate socket and destination.
2. Read TX head and tail.
3. Poll while the queue is full.
4. Select the current TX tail index.
5. Write metadata.
6. Copy payload.
7. Flush the cache range if necessary.
8. Execute an ordering barrier.
9. Advance TX tail.
10. Return immediately after enqueue.

It does not wait for:

* ARP resolution
* MAC transmission
* Remote reception

## 18.5 `udp_recv`

Sequence:

1. Validate socket, payload pointer, capacity, and info pointer.
2. Poll until RX head differs from RX tail.
3. Select current RX head entry.
4. Invalidate the entry’s cache range if necessary.
5. Execute an acquire/order barrier.
6. Read metadata.
7. If the user buffer is too small:

   * Return an error
   * Do not advance head
8. Copy payload.
9. Execute a release/order barrier.
10. Advance RX head.
11. Return payload length

# 19. Edge-case behavior

| Condition                              | Required behavior                                                  |
| -------------------------------------- | ------------------------------------------------------------------ |
| RX asynchronous FIFO overflow          | Terminate buffered prefix with an error marker, drop rest of frame |
| RX queue full                          | Consume and drop packet; do not start writer                       |
| No UDP socket matches                  | Consume and drop                                                   |
| Duplicate enabled port bindings        | Treat as invalid/no one-hot match and drop                         |
| Socket closed during RX DMA            | Finish/abort writer but suppress tail commit                       |
| IPv4 options present                   | Drop                                                               |
| IPv4 fragmented                        | Drop                                                               |
| Bad IPv4 checksum                      | Drop                                                               |
| UDP length inconsistent with IPv4      | Drop                                                               |
| UDP checksum nonzero                   | Accept without verifying, per scoped limitation                    |
| Ethernet FCS/MAC error                 | Suppress RX tail commit                                            |
| Frame truncated during payload         | Abort writer; suppress commit                                      |
| Extra Ethernet padding                 | Drain and ignore                                                   |
| AXI write error                        | Suppress RX tail commit                                            |
| TX metadata invalid                    | Drop descriptor and advance TX head                                |
| TX AXI read error                      | Abort FIFO frame and advance TX head                               |
| ARP cache miss                         | Send ARP request and wait                                          |
| ARP request for local IP               | Learn sender and transmit reply                                    |
| ARP reply for pending target           | Learn sender and resume UDP TX                                     |
| Unrelated ARP reply                    | Ignore or update only an existing cache entry                      |
| ARP timeout                            | Drop blocked TX descriptor and advance head                        |
| TX FIFO unavailable                    | Producer waits; no descriptor state is lost                        |
| Zero-byte UDP payload                  | Generate 28-byte IP/UDP payload; MAC pads Ethernet frame           |
| Destination outside subnet             | Resolve gateway MAC, preserve original IP destination              |
| Network configuration changed          | Flush ARP cache                                                    |
| Reset during frame                     | Reset FIFOs/engines and discard partial transactions               |
| Software changes DMA base while active | Unsupported; software must disable/empty queues first              |

# 20. Assertions worth adding

At minimum:

```text
RX tail advances only after:
    valid frame end
    and AXI write success
    and socket still enabled/matched

TX head advances only after:
    complete frame buffered
    or descriptor intentionally dropped

A producer never writes a full RX queue.

Software-visible queue pointers always contain valid indices.

Frame arbiter grant does not change before last/abort.

Two frame producers are never simultaneously accepted.

No partial TX buffer becomes READY after abort.

ARP resolved response corresponds to the currently pending IP.

IPv4 UDP payload bytes written never exceed 1472.

Parser always drains through frame last after any rejection.

MAC RX abort marker separates a truncated frame from the next frame.
```

# 22. Resource-conscious implementation choices

Use these choices to control LUT/FF growth:

* One shared AXI reader
* One shared AXI writer
* One UDP TX engine
* One UDP RX engine
* One outstanding ARP resolution
* Four-entry ARP cache
* Sequential IPv4 checksum accumulators
* Fixed 20-byte IPv4 headers
* Fixed 8-byte UDP headers
* Case-based 1500-byte address offsets
* No packet reassembly
* No UDP checksum hardware
* No per-socket TX engines
* No per-socket AXI writers
* No full-frame RX staging RAM unless the existing writer cannot support aborts

The protocol control logic should be modest relative to the existing BRAM-backed FIFO structures. The ARP cache is only a few hundred bits of state, and the checksum datapaths require small adders and counters.

# 24. Final architectural judgment

This architecture is sound for the Basys 3 and aligns with the agreed API and queue map.

The most important properties are:

* AXI-Lite state is separated from protocol execution.
* ARP is independent and can make progress while UDP TX is stalled.
* Ethernet framing remains centralized in the MAC.
* TX descriptors are released only after complete internal buffering.
* RX descriptors are published only after full frame validation and successful DMA completion.
* Queue states are unambiguous and all physical entries remain usable.
* Socket close cannot accidentally publish an in-flight packet.
* Invalid or unresolved packets cannot permanently block the engine.
* The design stays resource-conscious by sharing DMA engines and limiting concurrency.

The resulting system is not merely an Ethernet MAC with a parser attached. It is a small hardware IPv4/UDP endpoint with a coherent hardware/software ownership model, deterministic queueing, ARP-based next-hop resolution, and standard application-facing UDP semantics.
